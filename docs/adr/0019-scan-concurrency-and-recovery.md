# ADR-0019: 限制单机扫描并恢复完成报告

- **状态**: Accepted
- **日期**: 2026-07-11
- **相关**: ADR-0016

## 背景

平台为每个请求启动独立线程和模型子进程，但多个任务会争抢同一 GPU。任务字段也可能被执行线程、取消请求和查询请求并发读写。服务重启后，已经完整生成的报告仍在磁盘，却无法再通过任务 API 访问。

## 决策

1. `ScanManager` 使用有界 semaphore，单机默认同时运行一个扫描；其余任务保持 queued。
2. 每个 `ScanJob` 使用独立锁保护状态、日志、事件和进程引用。
3. queued/running 取消与进程启动采用同一状态锁，取消后的 queued 任务不得启动。
4. 服务启动时扫描 `results/platform/*.json`，只把可解析的完整 JSON 恢复为 completed 任务。
5. queued/running 任务不跨进程恢复，也不声称能够续跑。

## 理由

- 默认单并发符合当前单 GPU 演示环境，避免显存竞争导致两个任务同时失败。
- 完成报告是不可变产物，可以不引入数据库就安全恢复。
- 对运行中任务做伪恢复会丢失子进程、日志和取消语义，比明确中断更危险。

## 后果

- 单机 API 具备确定的排队、取消和 completed 报告恢复语义。
- 多 GPU、多进程或多实例部署仍需持久化任务队列、worker ownership 和租约/心跳机制。
